# Why Anneal

> **Status:** Current product thesis for the pre-1.0 implementation.
> Every command shown here exists in the CLI documented by the
> [README](../README.md#current-cli). Future capabilities are separated into the
> [roadmap](roadmap.md) and [proposals](proposals/README.md).

Anneal is for repositories that have outgrown independent language-specific build commands
but do not want to replace Cargo, pnpm, or other native tools with a completely new build
universe.

Its product boundary is deliberately narrow:

> Anneal coordinates native tools through an explicit cross-language graph, routes generated
> files between them, runs declared actions inside sandboxed boundaries, and reuses exact
> local results and selected native-tool state where the current contract allows.

Cargo still understands Cargo manifests and lockfiles. pnpm still performs pnpm installation
and scripts. Nickel still evaluates Nickel. Anneal owns the boundaries between those tools:
inputs, outputs, generated-file edges, execution policy, and reuse.

## The problem

Polyglot repositories tend to accumulate three kinds of uncertainty.

### The real inputs are implicit

A build may read an `.env` file, a global configuration file, an undeclared tool, or a package
outside the path globs used by CI. A cache that hashes only the inputs someone remembered to
list can reuse a result produced from a different effective input set.

### Generated files drift

When one language generates configuration, schemas, or source consumed by another, teams often
check the result into the repository and add a “regenerate and diff” CI step. The repository
then has two sources of truth and no graph edge explaining which consumers depend on the
generated bytes.

### Correct build systems can be expensive to adopt

A highly decomposed build graph can provide excellent correctness and reuse, but adopting it
may require replacing native manifests, workflows, and ecosystem semantics. Smaller teams may
reasonably prefer coarse native-tool units even when that sacrifices some cache granularity.

Anneal is an attempt to make that coarse starting point structurally sound and progressively
refinable.

## The core idea: enforce the boundary

Anneal does not assume a declared input list is trustworthy merely because it was written down.
A sealed action runs in a sandbox where undeclared host paths and network access are denied
according to its policy and platform enforcement grade.

This changes the failure mode of a coarse declaration:

- Declaring too much may rebuild more than necessary.
- Declaring too little should fail at the sandbox boundary instead of silently producing a
  reusable result from hidden inputs.

The guarantee is platform-graded. Linux sealed actions use namespace isolation and are graded
`Enforced`. macOS sealed actions use a deny-by-default Seatbelt policy and are graded
`LoudBestEffort`, not Linux-equivalent hermeticity. The exact boundary is documented in the
[sealed sandbox contract](sandbox-contract.md).

Sandboxing proves an input boundary; it does not prove output determinism. Rule authors and the
engine still need an explicit cacheability contract, and Anneal has pre-1.0 hardening work in
that area.

## Wrap native tools instead of replacing them

Anneal's first-party rules model native work at a deliberately coarse level.

- `cargo_workspace` wraps Cargo workspace compilation and test execution.
- `pnpm_workspace` performs a frozen offline install and runs declared scripts.
- `nickel_eval` exports a self-contained Nickel input in a selected format.
- Generic rules connect files, aliases, and commands across package boundaries.

The native tool remains authoritative for the semantics inside its unit. Anneal does not try to
reimplement Cargo's resolver or pnpm's linker as a new rule graph.

Coarse modeling is a performance tradeoff, not permission to read the whole machine. The action
still receives declared sources, generated inputs, toolchain roots, environment, and network
policy through an explicit contract.

## Generated files are graph artifacts

A generated file has a producer, content identity, and consumers. Downstream actions receive it
through a declared sandbox path rather than an ambient file that happens to exist in the
worktree.

That gives generated data two useful behaviors:

1. A source change rebuilds the generator and the consumers connected to its output.
2. If the generated bytes do not change, content identity can stop invalidation at that edge.

For IDEs and unwrapped native commands that require the file on disk, Anneal exposes an explicit
bridge:

```console
anneal materialize //path/to/package:consumer
```

Materialization mirrors the generated inputs routed to that consumer, tracks ownership and
digests, avoids needless mtime churn, and refuses destructive replacement by default.

Useful inspection and maintenance forms include:

```console
anneal materialize //path/to/package:consumer --check
anneal materialize //path/to/package:consumer --list
anneal materialize //path/to/package:consumer --clean
```

Materialization is not automatic staged analysis: if a generated manifest must reshape
Anneal's own target graph, the current workflow requires an explicit materialize-and-rerun step.

## Reuse has two layers

Anneal can avoid native work through an exact action-cache hit. It also keeps selected native
working state warm so a cache miss does not necessarily become a container-cold invocation.

For Cargo, that means preserving managed `target/` state and Cargo fingerprint reuse. Anneal
currently sets `CARGO_INCREMENTAL=0`, so it does not preserve rustc incremental code generation.
For pnpm, managed installation state can be restored for declared script consumers.

The honest performance contract is scenario-based:

> Anneal should win decisively when it avoids invoking the native tool, and should impose
> bounded overhead when it must invoke that same tool.

It is not credible to promise that every wrapped incremental miss will beat an already-warm
native command. Current methodology and measurements live in
[the benchmark record](benchmarks/current.md).

Warm working directories are local accelerators. Anneal does not treat Cargo `target/` or pnpm
`node_modules` snapshots as portable cross-machine cache artifacts.

## Current workflow

The supported contributor and adopter environment is Nix:

```console
nix develop
cargo run -p anneal-cli -- --help
```

First-party rules resolve tools through `ANNEAL_TOOLCHAIN_MANIFEST`, which the development shell
exports. Anneal does not currently scan the host `PATH`, provision toolchains, or provide an
`init` command.

The current build and test commands operate on explicit labels:

```console
anneal build //path/to/package:target
anneal test //path/to/package:target
```

CI can require the strongest enforcement grade available for its actions:

```console
anneal build //path/to/package:target --require-enforced
```

Dependency-impact and path queries are available separately:

```console
anneal affected --since origin/main
anneal why //app:binary //lib:generated_config
anneal why //app:binary --since origin/main
```

`affected --since` is graph-derived rather than a hand-maintained path filter, but currently
omits untracked files that have not been added to Git. The local focus-cone path used by
`build` and `test` derives dirty files from `git status` and does include untracked files.

## Where Anneal sits

Anneal is not trying to be a transparent wrapper around arbitrary undeclared shell commands,
and it is not currently trying to expose the complete extensibility or distributed-execution
surface of Bazel-class systems.

It chooses a middle tradeoff:

| Dimension | Anneal's current choice |
|---|---|
| Ecosystem semantics | Native tools and lockfiles remain authoritative |
| Initial graph granularity | Coarse first-party workspace rules |
| Cross-language dependencies | Explicit providers, artifacts, and action edges |
| Hidden inputs | Denied by the declared sandbox boundary, subject to platform grade |
| Reuse | Exact local action results plus selected warm native state |
| Generated files | Build artifacts with optional explicit worktree materialization |
| Distribution | Local only; no shared cache or remote executor |
| Toolchains | Closure-complete Nix manifest for the supported path |

The bet is that teams can accept conservative over-rebuilding during adoption more readily than
silent stale reuse or a wholesale ecosystem conversion.

## Current limits

Anneal should not yet be presented as a production-complete replacement for an established
organization-wide build platform.

Important current boundaries include:

- no remote/shared cache or remote execution;
- no store garbage collection or retention policy;
- no automatic onboarding or Anneal-managed tool provisioning;
- no general input-sensing or trust-explanation command;
- no Linux VM execution path on macOS;
- no demand-driven pruning for ordinary `build` and `test`;
- coarse first-party rule granularity and incomplete ecosystem coverage;
- known cache-identity and persistent-state hardening tasks listed in
  [`TODO.md`](../TODO.md).

Those limits do not erase the implemented vertical slice. They define the boundary within
which Anneal's current claims can be evaluated honestly.

## Direction

The next stage prioritizes correctness hardening, operational lifecycle, diagnostics, and
adoption over adding more speculative surface area. Shared caching, input sensing, a Linux VM,
and staged analysis remain possible later stages with explicit prerequisites.

See the [roadmap](roadmap.md) for that sequencing and the
[proposal index](proposals/README.md) for designs that are intentionally not current product
promises.
