# Simplex rules — package graphs, code generation, and pinned acquisition

> **Status: proposed, not implemented.** This document specifies a first-party Anneal
> ruleset for Blockstream Research's [Smplx](https://github.com/BlockstreamResearch/smplx)
> framework, pinned to the behavior of Smplx `v0.0.8`. It is a companion to
> [`docs/rules.md`](rules.md) (the general rule contract),
> [`docs/sandbox-contract.md`](sandbox-contract.md) (the action boundary), and
> [`TODO.md`](../TODO.md)'s rule, acquisition, and staged-analysis work. It is indexed as a
> proposal by [`docs/proposals/README.md`](proposals/README.md).

The ruleset wraps **`simplex build` only**. It does not wrap `simplex install`,
`simplex test`, `simplex regtest`, deployment, or wallet operations.

## 1. Decision summary

The ruleset has three user-facing rules and one typed provider:

| Rule | Purpose | Actions | Provider |
|---|---|---:|---|
| `simplex_package` | Describe a local package of `.simf` modules and its import graph | none | `SimplexPackageInfo` |
| `simplex_build` | Generate flattened contracts and Rust bindings for explicit entrypoints | one sealed build action | `FileSet` |
| `simplex_archive` | Acquire a public, hash-pinned remote Simplex package | one fixed-output fetch | `SimplexPackageInfo` |

The load-bearing decisions are:

1. **Package description and code generation are separate targets.** Dependencies need
   SimplicityHL sources and package metadata, not each dependency's generated Rust bindings.
   A package target therefore contributes no action; exactly the requested
   `simplex_build` target invokes Smplx.
2. **Every Simplex dependency is an Anneal label.** The edge is visible to loading,
   `affected`, `why`, cycle detection, and provider validation. A dependency's origin
   (local sources or a pinned archive) is hidden behind `SimplexPackageInfo`.
3. **Anneal synthesizes every `Simplex.toml`.** BUILD attributes are the graph source of
   truth. Smplx receives only sandbox-local `path` dependencies; Anneal never emits a
   native Smplx `git` entry and never runs `simplex install`.
4. **A generic `FileSet` is not a Simplex package.** Consumers require
   `SimplexPackageInfo` and reject a Cargo target, arbitrary `genrule`, or `filegroup`
   during analysis unless an explicit adapter packages its `.simf` files.
5. **Remote acquisition is byte-pinned.** `simplex_archive` requires an immutable archive
   URL and a SHA-256 of the downloaded bytes. Optional source-revision metadata aids review
   but is not presented as cryptographic provenance. The build action remains network-denied.
6. **Cacheability is earned separately from hermeticity.** The build is sealed from the
   first implementation, but stays non-action-cacheable until Smplx's generated bytes are
   made deterministic and the result passes Anneal's reproducibility gate.

## 2. Terminology and upstream behavior

This document uses:

- **SimplicityHL** for the `.simf` language and module system;
- **Smplx** for Blockstream Research's Rust framework and repository;
- **`simplex`** for the installed CLI binary;
- **Simplex package** for a package root containing a SimplicityHL source directory and
  dependency metadata.

Smplx's own README labels the framework extremely early and warns that breaking changes are
expected. Anneal therefore supports explicit, tested Smplx versions rather than a floating
“latest” toolchain; changing that version requires updating the output planner and its parity
fixtures in the same change.

### 2.1 What `Simplex.toml` contains

In Smplx `v0.0.8`, the root configuration has four sections:

| Section | Contents | Used by `simplex build` |
|---|---|---|
| `[build]` | `src_dir`, `simf_files`, `out_dir` | yes |
| `[dependencies]` | import name → exactly one of `path` or `git` | yes |
| `[regtest]` | mnemonic, funding, ports, RPC credentials | no |
| `[test]` | mnemonic, funding, verbosity, Esplora/RPC settings | no |

The build defaults in code are:

```toml
[build]
src_dir = "simf"
simf_files = ["**/*.simf"]
out_dir = "src/artifacts"
```

The CLI nevertheless deserializes and validates the **whole** root configuration before
dispatching `build`. An invalid test-network setting can therefore block code generation,
and treating the source file as a build input could admit irrelevant mnemonic or RPC data
into Anneal's CAS. A build-only normalized manifest avoids both failure modes.

The current CLI also ignores its parsed config-path field and reads a file named exactly
`Simplex.toml` from its working directory. The rule must provide that conventional path.

Sources:

- [root configuration](https://github.com/BlockstreamResearch/smplx/blob/v0.0.8/crates/cli/src/config/core.rs)
- [build configuration and defaults](https://github.com/BlockstreamResearch/smplx/blob/v0.0.8/crates/build/src/config.rs)
- [`simplex build`](https://github.com/BlockstreamResearch/smplx/blob/v0.0.8/crates/cli/src/commands/build.rs)
- [CLI dispatch](https://github.com/BlockstreamResearch/smplx/blob/v0.0.8/crates/cli/src/cli.rs)

### 2.2 Dependency resolution

A dependency name becomes the first component of a SimplicityHL import:

```toml
[dependencies]
merkle = { path = "deps/merkle" }
```

```text
use merkle::build_root::get_root;
```

The dependency root must itself be a Simplex package. Smplx reads its
`Simplex.toml`, finds that package's `[build].src_dir`, registers the directory under
the import name, then recursively reads the package's `[dependencies]`. A dependency's
`simf_files`, `out_dir`, test, and regtest settings do not participate in the consuming
build.

This recursive path does not deserialize the dependency's full CLI configuration: it parses
only `[build]` and `[dependencies]`. Malformed data in either relevant section fails, while
unrelated top-level test/regtest content is not validated there. Anneal still normalizes the
whole staged closure so source manifests cannot hide graph edges or alter package roots.

Files within the current package use `crate::...`; `[dependencies]` is for named package
roots. It does not select build entrypoints — `[build].simf_files` does that.

Sources:

- [dependency collector](https://github.com/BlockstreamResearch/smplx/blob/v0.0.8/crates/build/src/collector.rs)
- [source and remapping resolver](https://github.com/BlockstreamResearch/smplx/blob/v0.0.8/crates/build/src/resolver.rs)
- [upstream multi-dependency fixture](https://github.com/BlockstreamResearch/smplx/blob/v0.0.8/fixtures/simf/imports/multidep.simf)

### 2.3 Generated outputs

For each selected `.simf` file containing `fn main`, Smplx:

1. resolves and flattens its imports;
2. writes the flattened `.simf` below `out_dir/src_dir/`;
3. writes a Rust binding below `out_dir/`, mirroring the entrypoint's relative
   directory and deriving the module name from the filename;
4. writes `mod.rs` files for the binding directory hierarchy.

The output set is structured enough to derive from an explicit entrypoint list, but the
current CLI exposes no machine-readable build plan. Anneal must keep a small output planner
in parity with its supported Smplx version or, preferably, move to an upstream
`simplex build --plan-json` interface if one appears.

Source: [artifact generator](https://github.com/BlockstreamResearch/smplx/blob/v0.0.8/crates/build/src/generator.rs).

## 3. Goals and non-goals

### Goals

- Represent local Simplex packages as ordinary cross-package Anneal targets.
- Make every direct and transitive import edge visible in the Anneal graph.
- Generate the minimal `Simplex.toml` files Smplx expects without ingesting unrelated
  test/regtest configuration.
- Run `simplex build` sealed, offline, with a pinned toolchain and complete source closure.
- Expose generated Rust bindings and flattened contracts as a `FileSet` for
  `cargo_workspace(data = ...)` and other consumers.
- Reject incompatible dependencies before execution through a typed provider.
- Support public remote packages through hash-verified archives while presenting the same
  provider as a local package.
- Keep a half-converted or mixed-language repository coarse but correct.

### Non-goals

- `simplex install`, `test`, `regtest`, `clean`, deployment, or wallet SDK operations.
- Network access from `simplex build`.
- Native Smplx `git` dependency semantics.
- Private repositories, credentials, SSH agents, signed URLs, or ambient credential helpers.
- Automatic discovery of remote transitive dependencies from a fetched archive.
- Submodule initialization or Git LFS downloads.
- Inferring a Simplex package from an arbitrary `FileSet` or filename suffix.
- Making a user assert cacheability.
- Replacing Cargo's responsibility for compiling the generated Rust bindings.

## 4. User-facing rules

### 4.1 `simplex_package`

`simplex_package` describes a local package and emits no actions:

```python
simplex_package(
    name = "pkg",
    src_dir = "simf",
    deps = {
        "//simplex/math:pkg": "math",
    },
)
```

| Attribute | Type | Default | Meaning |
|---|---|---|---|
| `name` | implicit target name | required | Anneal target identity |
| `src_dir` | string | `"simf"` | package-relative SimplicityHL source root |
| `deps` | label-keyed string dict | `{}` | dependency target → import name |

The target's Anneal package directory is the native Simplex package root. One Anneal package
directory represents one Simplex package root; multiple native roots require nested/sibling
directories with their own `BUILD` files.

A descendant directory containing `BUILD` is an Anneal package boundary. The selective walk
stops before that directory, and `src_dir` itself may not cross such a boundary. Code below a
nested package must enter through that package's `SimplexPackageInfo` and an explicit import
edge; it cannot be consumed as the parent's `crate::` source. This keeps source ownership,
`affected`, and the bytes in the action key aligned.

The rule ingests every regular `.simf` file below `src_dir` conservatively; Smplx decides
which modules an entrypoint actually imports. Selection must happen **before** CAS ingestion:
filtering the result of Anneal's current whole-tree helper would already have persisted the
rejected files. Other files — especially a package-root `Simplex.toml` — never enter this
provider, the CAS, or the build action, even when `src_dir = "."`.
If SimplicityHL later gains auxiliary non-`.simf` inputs, they require an explicit typed
attribute rather than silently widening this tree. V1 rejects symlinks below `src_dir`
rather than following Smplx's permissive glob-walk behavior or silently omitting a linked
module. Repository and build-system internals such as `.git`, `.anneal`, and `target` are
always excluded, and no source may be staged inside the reserved `.anneal/simplex`
namespace.

The selective walker must be rooted at an open package-directory descriptor, inspect every
`src_dir` ancestor and entry without following links, and verify containment before
ingestion. A symlink used as `src_dir` or as one of its ancestors is rejected just like a
symlink inside the tree. Validation followed by `Cas::ingest_file(path)` is not sufficient:
that API reopens the pathname and leaves a symlink-swap race. The walker uses
descriptor-relative no-follow opens, verifies a regular file with `fstat`, and passes the
already-open file to a new CAS ingestion API (or reads it for `Cas::put`).

All accepted path components must be valid UTF-8 and unique under a portable path key:
Unicode NFC followed by default case folding, applied component by component. This
conservative rule prevents lossy action-key encoding and case/normalization overwrites when
the same graph moves between filesystems. The Unicode data version is pinned as part of the
rule/driver contract and covered by path-collision fixtures.

`deps` is written `dependency label -> Simplex import name` because Anneal's
`LabelKeyedStringDict` turns every key into a real graph edge. The provider stores the same
relationship internally as `import name -> package key`.

`simplex_package` returns `SimplexPackageInfo` and an empty action list. It does **not**
expose the package sources as its default `FileSet`: consumers needing Simplex semantics
must request the typed provider, while unrelated file routing should use an explicit
`filegroup`.

### 4.2 `simplex_build`

`simplex_build` invokes the compiler for one package:

```python
simplex_build(
    name = "artifacts",
    package = "//apps/vault:pkg",
    contracts = [
        "vault.simf",
        "policies/recovery.simf",
    ],
    out_dir = "src/artifacts",
)
```

| Attribute | Type | Default | Meaning |
|---|---|---|---|
| `name` | implicit target name | required | Anneal target identity |
| `package` | label | required | target providing `SimplexPackageInfo` |
| `contracts` | string list | required, nonempty | exact entrypoints relative to the package's `src_dir` |
| `out_dir` | string | `"src/artifacts"` | output root relative to the build target's Anneal package |

Exact `contracts` replace an opaque `simf_files` glob. The rule needs the set before
execution to declare outputs; the package's entire source tree still enters the action key
so imports are complete. Contract paths must be normalized, relative, unique, free of glob
metacharacters, and contained by `src_dir` after staging. Directory and filename components
that Smplx cannot convert to valid Rust modules are rejected before execution.

`contracts` order is not semantic. Analysis rejects exact or portable-path duplicates, then
sorts the normalized paths before serializing `simf_files`, planning outputs, and creating
the action. Reordering a BUILD list therefore does not change the manifest or action key.

`package` may name either a local `simplex_package` or a remote `simplex_archive`; both are
accepted through `SimplexPackageInfo`. Regardless of origin, the selected root package's
complete `.simf` tree is materialized at the build action's working directory, while only
its dependencies use the hidden package staging area. For an archive root, safe extraction
places its declared `subdir/src_dir` at `<cwd>/src_dir` before the root manifest is installed.

`out_dir` belongs to the **build target's** Anneal package, not necessarily the referenced
package target's directory. This is where its `FileSet` is routed/materialized for consumers.
It and every planned output must stay outside Anneal/repository/mutable-build namespaces:
no path component may be `.anneal`, `.git`, or `target`. Every planned file must also be
disjoint from declared source inputs and pre-existing workspace sources.

The build target exposes every declared generated file through its default `FileSet`. A
Cargo consumer can route them into the native source layout:

```python
cargo_workspace(
    name = "rust",
    data = ["//apps/vault:artifacts"],
)
```

### 4.3 `simplex_archive`

`simplex_archive` describes and acquires a public remote package:

```python
simplex_archive(
    name = "merkle",
    url = "https://github.com/acme/merkle/archive/<full-commit>.tar.gz",
    source_revision = "<full-commit>",
    sha256 = "<sha256-of-downloaded-archive-bytes>",
    strip_prefix = "merkle-<full-commit>",
    subdir = ".",
    src_dir = "simf",
    deps = {
        "//third_party/simplex:math": "math",
    },
)
```

| Attribute | Type | Default | Meaning |
|---|---|---|---|
| `name` | implicit target name | required | Anneal target identity |
| `url` | string | required | public immutable archive URL |
| `source_revision` | string | absent | optional human-facing source revision metadata |
| `sha256` | string | required | SHA-256 of the exact downloaded archive bytes |
| `strip_prefix` | string | required | normalized single archive-root prefix to remove |
| `subdir` | string | `"."` | Simplex package root within the stripped archive |
| `src_dir` | string | `"simf"` | source root relative to `subdir` |
| `deps` | label-keyed string dict | `{}` | explicit dependency target → import name |

There is no open-ended `format` attribute in v1: the only accepted payload is the
gzip-compressed tar format defined in §10.3. URL suffixes are advisory and do not select the
parser.

`source_revision` and `sha256` serve different purposes. A full immutable object ID helps a
reviewer identify the intended Git source, but the generic archive rule cannot prove that the
server's bytes came from that object. It is explicitly informational. The SHA-256 is the
actual Anneal enforcement boundary: a mismatch fails closed and is never auto-updated.
For Git-hosted archives the URL should name a full commit, never a branch or tag; immutable
non-Git release assets need no `source_revision`.

`simplex_archive` produces one fixed-output fetch action and the same
`SimplexPackageInfo` as `simplex_package`. It does not run `simplex build` and does not
expose Smplx's native `git` form.

Longer-term, `simplex_archive` should become thin package-typing sugar over generic
`http_archive` / `fetch_git_tree` acquisition. It is acceptable to implement the first
consumer locally as long as the provider boundary does not bake downloading into
`simplex_build`.

## 5. Complete local example

```python
# simplex/math/BUILD
simplex_package(
    name = "pkg",
    src_dir = "simf",
)
```

```python
# simplex/merkle/BUILD
simplex_package(
    name = "pkg",
    src_dir = "simf",
    deps = {
        "//simplex/math:pkg": "math",
    },
)
```

```python
# apps/vault/BUILD
simplex_package(
    name = "pkg",
    src_dir = "simf",
    deps = {
        "//simplex/merkle:pkg": "merkle",
    },
)

simplex_build(
    name = "artifacts",
    package = "//apps/vault:pkg",
    contracts = ["vault.simf"],
    out_dir = "src/artifacts",
)

cargo_workspace(
    name = "rust",
    data = ["//apps/vault:artifacts"],
)
```

The target graph is:

```text
//simplex/math:pkg
        ↑ import `math`
//simplex/merkle:pkg
        ↑ import `merkle`
//apps/vault:pkg
        ↑ package
//apps/vault:artifacts
        ↑ generated data
//apps/vault:rust
```

Changing Math affects Merkle, the Vault package, its generated artifacts, and the Cargo
consumer through ordinary reverse dependency closure. A diamond dependency is analyzed once
and staged once because package identity is independent of the import aliases on incoming
edges.

For the most precise ownership and `affected` behavior, each native Simplex package should
have a `BUILD` file at its package root. A Simplex provider target may coexist with other
targets in that Anneal package; consuming package-owned files is shared usage, not a second
workspace owner.

## 6. Typed provider model

The current `ProviderSet` contains only `FileSet`. This ruleset adds a distinct channel:

```rust
struct ProviderSet {
    files: Option<FileSet>,
    simplex_package: Option<SimplexPackageInfo>,
}

struct SimplexPackageInfo {
    root: SimplexPackageKey,
    packages: BTreeMap<SimplexPackageKey, SimplexPackageNode>,
}

struct SimplexPackageNode {
    origin: Label,
    src_dir: PathBuf,
    source: SimplexPackageSource,
    deps: BTreeMap<String, SimplexPackageKey>, // import name -> package
}

enum SimplexPackageSource {
    Local { files: FileSet },
    Archive {
        archive: Artifact,
        strip_prefix: PathBuf,
        subdir: PathBuf,
    },
}
```

The exact Rust representation may differ, but these invariants are required:

- `root` identifies the package requested by this provider.
- `packages` is a closed, flattened transitive package graph.
- A package key derives stably from the producing target label, not from an edge alias.
- Import aliases are parent-edge metadata. The same target may be named `math` by one
  parent and `arithmetic` by another without duplicating package identity.
- Local sources retain content-addressed `Artifact` references.
- Remote sources retain the verified archive `Artifact` output reference and extraction
  metadata; analysis never pretends the archive's interior files already exist.
- Merging a diamond with identical nodes deduplicates it. The same key with conflicting
  metadata is an analysis error.

Each `simplex_package` or `simplex_archive` receives its direct dependencies after they
have been analyzed, verifies their `SimplexPackageInfo`, merges their closed package maps,
then inserts its own node. The existing analyzer already detects target cycles before a
provider can be constructed.

### 6.1 Why `FileSet` is insufficient

A list of `.simf` files does not say:

- which directory is the package root;
- what its `src_dir` is;
- which import name maps to which child package;
- what transitive packages must be staged;
- whether the bytes are local files or a pinned archive awaiting extraction.

Accepting a generic `FileSet` would postpone a type error into an opaque Smplx failure and
make the dependency graph incomplete. The consumer instead fails during analysis:

```text
simplex_package //apps/vault:pkg:
dependency //config:data, imported as `math`, does not provide
SimplexPackageInfo; expected a target providing SimplexPackageInfo
```

An `alias` remains valid because it forwards the complete provider set. A future explicit
adapter may wrap generated `.simf` outputs in `SimplexPackageInfo`; generated package inputs
are unsupported until that adapter (or an explicit `srcs` extension) exists. There is no
implicit suffix-based conversion.

## 7. Analysis semantics

### 7.1 `simplex_package`

Analysis:

1. validates `src_dir` and import aliases;
2. verifies every `deps` label provides `SimplexPackageInfo`;
3. uses a pre-ingest filtered source walk to ingest only regular `.simf` files below
   `src_dir`;
4. merges dependency closures and detects conflicts;
5. inserts the local package node;
6. returns no actions and one `SimplexPackageInfo`.

The dict is written `label -> import name` because `LabelKeyedStringDict` makes every key a
real loader edge. The provider stores it in the more natural `import name -> package key`
form. Duplicate import names, invalid SimplicityHL identifiers, and reserved names are clean
analysis errors.

This requires a selective source helper (for example,
`RuleContext::source_tree_filtered`). Calling `source_tree` and filtering its returned
artifacts is not equivalent: the current helper has already put every regular file into the
CAS by then. The helper must use no-follow traversal and containment checks; wrapping the
current `read_dir(base)` implementation is insufficient because `base` itself may resolve
through a symlink. It also requires CAS ingestion from an already-open file so validation and
content hashing are one descriptor-based operation rather than a check-then-reopen sequence.

### 7.2 `simplex_archive`

Analysis performs the same dependency-provider checks and closure merge, then emits a
single fixed-output fetch action whose output is the archive artifact. The node carries that
output reference rather than a fabricated file tree. `sha256` must parse as an Anneal
`Digest`. V1 accepts HTTPS only and rejects URL userinfo, fragments, and user-supplied query
strings; the shared fetch path must also enforce redirect, timeout, byte-limit, redaction,
and network-destination policy described in §10.3.

The fetch action:

- has no varying source inputs;
- uses a pinned hardened fetch helper, with the validated URL passed as one argv element;
- permits network only because `fixed_output(expected)` fences the result;
- declares exactly one regular-file output;
- is platform-independent;
- cache-hits by expected content digest before touching the network.

The current shell-string `curl` precedent is not sufficient for this rule. Remote support is
gated on the hardened helper described in §10.3; it must enforce destination policy itself
and must not interpolate URLs into `sh -c`.

The archive artifact must remain an `ArtifactSource::Output` from this action even though
its expected digest is already known. Replacing it with a bare source digest during analysis
would erase the execution edge and fail on a cold CAS where the fetch still has to run.

### 7.3 `simplex_build`

Analysis:

1. verifies `package` provides `SimplexPackageInfo`;
2. validates exact contract paths and `out_dir`;
3. plans every expected output for the pinned Smplx version;
4. rejects generated/source and generated/generated collisions;
5. creates canonical manifests for the root and every package node;
6. emits one build action consuming the full package closure;
7. exposes planned outputs as a `FileSet`.

No execution-time discovery may add packages or outputs to the graph. An undeclared import
or output-shape mismatch fails loudly; it never widens the sandbox or silently escapes the
action key.

## 8. Synthesized manifests and sandbox layout

For the local example, the build sandbox is:

```text
<cwd>/
├── Simplex.toml                         # synthesized root manifest
├── simf/                                # complete root .simf tree (abridged here)
│   ├── vault.simf
│   └── imported_helpers.simf
├── src/artifacts/                       # declared outputs
└── .anneal/simplex/
    ├── plan.json                        # canonical driver data, never shell source
    ├── archives/                        # verified archive inputs
    ├── manifests/                       # read-only canonical manifest blobs
    └── packages/
        ├── <merkle-key>/
        │   ├── Simplex.toml             # copied from canonical blob after staging
        │   └── simf/...
        └── <math-key>/
            ├── Simplex.toml
            └── simf/...
```

Package keys use a stable label-derived encoding suitable for a path (for example, a readable
prefix plus a digest of the canonical label). They are not import aliases and therefore do
not collide when two parents use different aliases.

The root manifest contains only build-relevant data:

```toml
[build]
src_dir = "simf"
simf_files = ["vault.simf"]
out_dir = "src/artifacts"

[dependencies]
merkle = { path = ".anneal/simplex/packages/<merkle-key>" }
```

Merkle's manifest is similarly synthesized:

```toml
[build]
src_dir = "simf"
simf_files = []
out_dir = ".anneal-unused"

[dependencies]
math = { path = "../<math-key>" }
```

Smplx's recursive collector reads only `src_dir` and `dependencies` from a dependency
manifest, so the unused build fields are explicit normalization, not behavior. Test and
regtest sections are never emitted.

### 8.1 Generated-file primitive

The manifest and driver-plan bytes are known during analysis. They should enter the CAS as
ordinary immutable blobs through a small rule-context helper, conceptually:

```rust
ctx.generated_file(path, bytes) -> Artifact
```

The helper:

- validates the logical relative path;
- calls `Cas::put` on the canonical bytes;
- returns a concrete blob artifact;
- does not record the path as a workspace source.

The action declares those blobs as inputs, so their content digests and destinations enter
the action key. It is unnecessary and less transparent to create a separate “echo this
manifest” action or embed multiline TOML in a shell command.

Archive packages require one staging wrinkle: an archive may itself contain
`Simplex.toml`, while declared input blobs are read-only. Canonical manifests therefore enter
at `.anneal/simplex/manifests/<key>.toml`; after safely extracting into a fresh writable
package directory, the action replaces any archive manifest with a copy of the canonical
blob. The archive can never choose the manifest Smplx reads.

### 8.2 Staging order

The build action uses a pinned `anneal-simplex-driver` and performs, in order:

1. read a canonical, generated staging plan and create every package root and declared
   `src_dir`, including empty dependency source directories that have no file artifact;
2. materialize a local selected root's `.simf` artifacts at `<cwd>/src_dir`, and local
   dependency artifacts below their hidden package roots;
3. safely unpack each archive into its own fresh scratch root, validate the selected
   `subdir/src_dir` subtree, then copy only accepted `.simf` files into the selected root's
   `<cwd>/src_dir` view or a dependency's hidden package view;
4. copy canonical `Simplex.toml` blobs into the root and every dependency package;
5. invoke `simplex build` from the pinned Simplex toolchain;
6. compare the files beneath `out_dir` with the analysis-time output plan;
7. return so Anneal can capture every declared output file.

Only the fixed-output fetch actions have network capability. Staging, extraction, and
`simplex build` are sealed and network-denied.

### 8.3 No shell interpolation

User-controlled values — including URLs, archive paths, `src_dir`, `out_dir`, contracts, and
planned output names — must never be interpolated into `sh -c` or another shell program.
The build driver receives paths through the parsed canonical staging plan, and the hardened
fetch helper receives the URL as one structured argv element. Canonical TOML is serialized by
a TOML library, not string concatenation. Both helpers repeat boundary/path validation before
filesystem mutation as defense in depth.

The driver and fetch helper are pinned toolchain inputs whose identities enter their action
keys. Hard-coded internal subprocess argv is acceptable; generated shell source containing
BUILD attributes is not.

## 9. Output contract

Let an entrypoint relative to `src_dir` be `dir/name.simf`, and let `module(name)` be
Smplx's pinned filename-to-Rust-module normalization. In `v0.0.8` the binding name comes
from `Path::file_prefix()`: Smplx lowercases it, maps non-alphanumeric/non-underscore
characters to `_`, collapses and trims repeated underscores, prefixes a leading digit with
`_`, then rejects Rust keywords or remaining invalid identifiers. Directory components are
passed through as Rust module identifiers and therefore require the same keyword/identifier
validation. The output planner declares:

- flattened source: `out_dir/src_dir/dir/name.simf`;
- Rust binding: `out_dir/dir/module(name).rs`;
- `out_dir/mod.rs`;
- one `mod.rs` for each ancestor directory below `out_dir/dir/`.

Shared `mod.rs` paths are declared once. Two entrypoints whose normalized module names or
paths collide are rejected during analysis. Logical output names should derive canonically
from their paths so they remain unique and stable.

Sources: [contract-name normalization](https://github.com/BlockstreamResearch/smplx/blob/v0.0.8/crates/build/src/macros/parse.rs) and
[artifact generation](https://github.com/BlockstreamResearch/smplx/blob/v0.0.8/crates/build/src/generator.rs).

The rule requires every contract to end in `.simf`. It also reproduces Smplx's directory
contract during analysis: `src_dir` and `out_dir` are normalized package-relative directory
paths, their final components may not be file-like paths with extensions under the supported
Smplx version, `src_dir` must exist for a local package, and `out_dir` must not be an existing
source file. These checks turn Smplx's late `resolve_local_dir` failures into rule-boundary
diagnostics.

For local packages, analysis verifies that each contract source exists but does not embed a
second SimplicityHL parser in v1. Smplx performs the authoritative parse during execution; a
file without `fn main` is filtered out, and the exact output auditor reports the corresponding
contract as producing no outputs. Remote package existence and entrypoint checks necessarily
use that same execution-time path because archive contents are unavailable to the current
single-pass analyzer.

Anneal's current CAS captures individual files, not directory trees. Static output planning
therefore makes the first rule possible without tree artifacts, but it creates a maintenance
obligation. Required safeguards are:

- fixtures covering flat, nested, multi-contract, and colliding names;
- parity tests against every supported Smplx version;
- a hard version pin in the toolchain identity;
- a small output-plan validator in the action that fails on an extra or missing generated
  file (the executor already detects missing declared outputs, but does not reject unrelated
  scratch files by itself).

Anneal's materializer pre-creates declared output parents, so the validator requires a fresh
output tree that is empty aside from that directory skeleton. Analysis rejects **any**
declared input path beneath `out_dir`, not merely an input colliding with a planned file.
After Smplx runs, the validator recursively lists only regular files without following links
and compares the normalized relative-path set exactly with the analysis-time plan. Symlinks,
special files, and any extra path fail the action before output capture.

Before emitting the action, `simplex_build` must also probe/record every planned workspace
output path (or reject an existing path directly). Anneal's global collision validator sees
source paths observed during analysis, not every unvisited file in the workspace; output
planning must not assume an existing `src/artifacts/mod.rs` will be discovered elsewhere.
That probe must inspect every ancestor without following links, reject dangling links, and
prove the destination remains under the build target's package directory; using the current
`package_file_exists` helper alone is insufficient.

An upstream machine-readable output plan or first-class Anneal tree artifacts would reduce
that maintenance burden, but neither may weaken generated-path collision checks.

## 10. Remote package acquisition

### 10.1 Why native Smplx `git` is rejected

Smplx `v0.0.8` accepts only a URL for a git dependency. `simplex install` performs a
shallow clone of the current default branch, records no revision or checksum, treats an
existing nonempty directory as installed, and writes no lockfile. That is incompatible with
a reproducible, content-addressed build.

Source: [`simplex install`](https://github.com/BlockstreamResearch/smplx/blob/v0.0.8/crates/cli/src/commands/install.rs).

Anneal acquires the source itself, verifies it, and rewrites the dependency to a local path.
The build never runs Git.

### 10.2 Explicit remote transitive dependencies

All direct and transitive dependencies of an archive package remain explicit Anneal labels:

```python
simplex_archive(
    name = "merkle",
    # url/sha256/strip_prefix omitted here
    deps = {
        "//third_party/simplex:math": "math",
    },
)
```

Anneal must not fetch the archive and then honor its embedded `git` entries. The archive's
manifest does not exist until execution, too late to change the analyzed target graph. Doing
so would require the staged-graph mechanism described in [`TODO.md`](../TODO.md) and a lock
format carrying immutable revisions and content hashes.

Explicit metadata makes the Anneal graph the lockfile. It also means local → remote,
remote → local, and remote → remote edges are the same provider composition. The synthetic
manifest for every node contains only resolved `path` entries.

### 10.3 Safe fetch and extraction

Fixed-output verification proves what bytes arrived; it does not make the network request
or those bytes harmless. The remote phase therefore includes both fetch hardening and archive
hardening.

The fetch path must:

- accept HTTPS URLs only and reject userinfo, fragments, and query strings in the declared
  URL;
- use fail-on-HTTP behavior, bounded redirects, connection/overall timeouts, retries, and a
  maximum downloaded byte count;
- permit redirects only to HTTPS and redact redirect URLs/query parameters from logs;
- reject every non-globally-routable destination (including loopback, link-local, private,
  multicast, and unspecified addresses) at every resolution and redirect step, connect only
  to an address from the validated resolution while preserving TLS hostname verification,
  and ignore ambient proxy/credential configuration;
- validate TLS with a pinned CA bundle included in the fetch-helper toolchain identity.

This last requirement is not supplied by a checksum: a pinned response can still be used for
SSRF, and Anneal's network-capable sandbox may reach host services. Anneal already has a native
Rust/TLS fixed-output fetch path with retries, expected-digest verification, and CAS reuse;
`simplex_archive` remains unavailable until that path is extended with the URL, redirect,
address, resolver, credential, and size policy above.

The Linux network-capable sandbox must also provide a functional, minimal resolver view.
Today its synthetic `/etc` contains only identity files, so ordinary hostname resolution is
not reliable. Before the remote phase, Anneal must either expose sanitized resolver/hosts
configuration (without mounting the host's whole `/etc`, and without copying an unreachable
loopback stub) or give the helper its own pinned resolver configuration. HTTP connections
remain restricted to the globally routable addresses validated from the answer; DNS
infrastructure access is not permission to connect HTTP to a private result.

A matching archive hash proves immutability, not safety. V1 accepts exactly one archive
format: a gzip-compressed POSIX tar stream, identified and parsed from its content rather
than a URL suffix. A pinned extraction library/helper is the security boundary; a raw
`tar x` command is not.

The helper must:

- require relative, normalized `strip_prefix`, `subdir`, and `src_dir` values;
- materialize only entries beneath `strip_prefix/subdir/src_dir` into the staged package's
  `src_dir`; files elsewhere remain covered by the archive digest but are absent from the
  package view Smplx can read;
- reject non-UTF-8 names, absolute paths, `..`, prefix escapes, devices, FIFOs, sockets,
  hard links, and symlinks;
- reject duplicate destinations and paths colliding under the same NFC-plus-case-folded
  portable key used for local sources;
- enforce a compressed-input byte limit **before** parsing, even when the archive came from
  a pre-existing fixed-output CAS hit, plus entry-count and expanded-byte limits against
  archive bombs;
- ignore or normalize ownership, timestamps, setuid/setgid bits, and irrelevant modes;
- verify `subdir` and `subdir/src_dir` exist;
- extract into a fresh dedicated scratch directory, then copy validated `.simf` entries into
  a collision-checked package view;
- write the synthesized manifest only after extraction.

Source archives are not recursive Git checkouts. V1 does not initialize submodules or fetch
Git LFS objects. A package must use a self-contained release archive, vendor those contents,
or model each external package as another pinned target. Detecting an LFS pointer under
`src_dir` should produce a targeted diagnostic where practical.

### 10.4 Archive stability and future tree hashing

A raw archive SHA-256 pins exact compressed bytes. A hosting provider may recompress an
otherwise identical source archive, which yields a safe availability failure rather than
silent drift. GitHub documents this distinction in its
[source-archive stability guidance](https://docs.github.com/en/repositories/working-with-files/using-files/downloading-source-code-archives#stability-of-source-code-archives).
Prefer immutable release assets or byte-stable mirrors when available; updating the hash
always requires review.

A future `fetch_git_tree` can fetch a full commit, normalize the extracted tree, and verify a
canonical tree digest independent of archive compression. Once Anneal has semantic tree
artifacts, extraction can become a cached producer rather than repeated staging inside each
consumer.

### 10.5 Private repositories

Authenticated archives are out of scope. Anneal has no secret input/mount abstraction, and
credentials in URLs, argv, or environment values could persist in action keys or logs.
`simplex_archive` rejects URL userinfo, query strings, and fragments and does not consult
ambient Git, proxy credentials, or credential-helper state. Private packages must be vendored
until a first-class secret capability exists.

## 11. Hermeticity, configuration, and caching

### `simplex_package`

- emits no action;
- consumes no universal axes;
- exposes source digests and typed metadata through its provider.

### `simplex_archive`

- emits one fixed-output action;
- is platform-independent;
- uses the archive SHA-256 as the output/cache identity;
- permits network only on that action;
- exposes no mutable state or snapshot;
- does not treat a CAS hit as archive-safety validation; the build driver rechecks compressed
  size and archive structure before extraction.

### `simplex_build`

- consumes no universal Anneal axes;
- runs sealed and network-denied;
- declares the pinned Simplex toolchain and `anneal-simplex-driver`;
- has no incremental state and no snapshot;
- is initially platform-sensitive;
- is initially `NonCacheable` while still capturing outputs for downstream actions.

Anneal's current action digest does not include the declared output map. Before this rule can
graduate to deterministic action caching, the general digest must hash each declared logical
output name and normalized path. Otherwise an output-planner change could reuse a result
whose stored output set belongs to the previous contract. The rule stays `NonCacheable`
until that substrate fix and the byte-reproducibility gate both land.

Smplx `v0.0.8` currently feeds unsorted `globwalk` results into a `Vec`, preserves that
discovery order for files, and stores child directories in a `HashMap`; both directly control
the order written to generated `mod.rs` files. Root-file and nested-directory module order can
therefore vary across processes. Before deterministic action caching is enabled, one of these
must happen:

1. upstream generation is made deterministic by sorting files/directories; or
2. the wrapper applies a narrowly justified deterministic normalization.

Sources: [entrypoint collection](https://github.com/BlockstreamResearch/smplx/blob/v0.0.8/crates/build/src/resolver.rs) and
[binding-tree generation](https://github.com/BlockstreamResearch/smplx/blob/v0.0.8/crates/build/src/generator.rs).

The result must then pass repeated cold builds in separate processes on every supported host.
That licenses `Deterministic` caching while retaining platform-sensitive action keys.
Declaring the rule platform-independent is a later, separate gate: generated bytes must also
compare equal across platforms, including path separators embedded in Rust bindings. Both
graduations are system/rule release decisions, never BUILD attributes.

## 12. Diagnostics and validation

Validation belongs at the earliest phase with enough information:

| Condition | Phase | Result |
|---|---|---|
| wrong attribute type / malformed label | load | schema error with BUILD location |
| absolute or escaping `src_dir`, `out_dir`, contract, `strip_prefix`, or `subdir` | load/analysis | rule error |
| duplicate or invalid import alias | analysis | rule error naming both edges |
| dependency lacks `SimplexPackageInfo` | analysis | typed-provider error |
| closure contains conflicting package nodes | analysis | provider merge error |
| target dependency cycle | analysis | existing cycle diagnostic |
| symlinked, non-UTF-8, or escaping local source path | analysis | contained-source error |
| local contract missing | analysis | source error |
| normalized output collision | analysis | generated-path collision error |
| internal output namespace, input beneath `out_dir`, or unsafe ancestor | analysis | path-contract error |
| invalid SHA-256 | analysis | rule error |
| non-HTTPS or credential-bearing URL | load/analysis | remote-source error |
| archive hash mismatch | execution | fixed-output mismatch; fail closed |
| wrong/oversized archive or unsafe entry / missing `src_dir` | execution | extraction error naming entry/package |
| contract lacks `fn main` or remote contract is missing | execution | targeted build/output error |
| undeclared import | execution | sealed Smplx failure; no network fallback |
| Smplx output missing or undeclared | execution | output-contract error |

Error messages should name the consuming target, dependency label, import alias, and expected
provider or path wherever applicable. A useful incompatible-provider diagnostic is:

```text
simplex_package //apps/vault:pkg:
dependency //config:data, imported as `math`, does not provide
SimplexPackageInfo; expected a target providing SimplexPackageInfo
```

## 13. Native CLI interoperability

BUILD attributes are the authoritative Anneal dependency graph. The sandbox does not pass a
checked-in root `Simplex.toml` through unchanged, because it may:

- hide cross-package edges from Anneal;
- contain moving native `git` dependencies;
- include irrelevant test/regtest credentials;
- invalidate code generation for unrelated test configuration changes;
- point at paths that do not exist in Anneal's staged layout.

Repositories may keep source manifests for direct `simplex` use, but the initial ruleset
ignores them inside the build action. Drift between native and Anneal graphs is therefore an
ergonomic limitation, not a cache-correctness risk.

Two compatible follow-ups are possible:

1. materialize/export Anneal's canonical manifests for native CLI use; or
2. add a mutually exclusive compatibility mode that reads a source manifest, validates it
   against label-typed BUILD dependencies, and injects only a normalized `[build]` plus
   `[dependencies]` projection.

The whole source manifest must never become a blind build input.

## 15. Rollout plan

### Phase 1 — local, dependency-free vertical slice

- add package-boundary-aware, descriptor-relative filtered source ingestion and CAS support;
- add `RuleContext::generated_file`;
- add the pinned Simplex toolchain;
- add the pinned `anneal-simplex-driver`;
- add `SimplexPackageInfo` and `simplex_package`;
- add `simplex_build` with exact contracts and static output planning;
- audit the generated file set exactly before capture;
- synthesize a root manifest with no dependencies;
- keep the build non-cacheable and platform-sensitive;
- route generated bindings into a Cargo fixture.

### Phase 2 — local package graph

- implement provider closure merge;
- synthesize dependency manifests and stable package staging paths;
- cover chains, diamonds, aliases, cycles, and incompatible provider errors;
- prove `affected` and `why` traverse Simplex import edges.

### Phase 3 — pinned remote packages

- implement `simplex_archive` using fixed-output fetch;
- harden HTTPS fetching and redirect/destination policy;
- provide functional sanitized DNS resolution to Linux network-capable actions;
- add safe, gzip-tar-only archive extraction with compressed and expanded limits;
- require explicit transitive dependency labels;
- cover hash mismatch, cache hit, archive traversal, subdir, and mixed local/remote graphs;
- keep authenticated sources out of scope;
- keep the consuming build action non-cacheable; remote acquisition does not depend on
  cache graduation.

### Phase 4 — reproducibility gate

- include the declared output map in Anneal's general action digest;
- fix upstream ordering or normalize it narrowly;
- run repeated cold builds in separate processes on every supported host;
- enable deterministic, platform-sensitive action caching only after the output-identity and
  same-platform reproducibility gates pass;
- compare outputs across platforms and clear platform sensitivity only if their bytes match;
- document the supported Smplx version/update procedure.

### Phase 5 — generalization and ergonomics

- extract generic `http_archive` / canonical `fetch_git_tree`;
- add semantic tree artifacts so archives unpack once;
- add native manifest export/materialization if direct CLI workflows require it;
- consider staged analysis only if an upstream lockfile justifies automatic remote closure
  discovery;
- consider a typed adapter for generated `.simf` package sources.

## 16. Test plan

Minimum coverage:

- schema defaults and path validation for all three rules;
- canonical TOML serialization and stable dependency ordering;
- source manifests containing test credentials never enter the CAS or action;
- unrelated non-`.simf` files below `src_dir` never enter the CAS or action;
- a parent source walk stops at descendant `BUILD` package boundaries, and contracts cannot
  cross them;
- symlinked `src_dir` roots/ancestors and symlinks inside local source trees are rejected;
- descriptor-based ingestion resists a validation-to-open symlink swap;
- non-UTF-8 paths and NFC/case-folded local, archive, and output collisions are rejected;
- dependency-free local build;
- an empty local dependency `src_dir` is created and canonicalizable in the sandbox;
- local direct, transitive, and diamond imports;
- the same dependency imported under different aliases by different parents;
- cross-package loading, `affected`, and `why` paths;
- alias target forwarding `SimplexPackageInfo`;
- rejection of `filegroup`, `genrule`, Cargo, and other non-Simplex providers;
- contract filename normalization and all expected output paths;
- contract-list reordering produces identical canonical manifests and action identity;
- duplicate normalized module/output collision rejection;
- nested contract directories and shared `mod.rs` declaration;
- missing `fn main` / missing output failure;
- rejection of extra, symlink, and special-file outputs from the exact output audit;
- rejection of any staged input beneath `out_dir`, internal `.anneal`/`.git`/`target`
  outputs, and symlinked output ancestors;
- sealed build cannot read undeclared workspace files or reach the network;
- source edit changes the build action key;
- dependency edit invalidates every true dependent and no unrelated target;
- fixed-output archive first fetch, CAS hit without network, and checksum mismatch;
- a cold Linux fetch resolves a public hostname through the sanitized resolver view;
- HTTPS-only URL validation plus userinfo/query/fragment, unsafe redirect, and
  private-destination rejection;
- URL/path shell metacharacters remain inert structured data and never execute;
- safe-extraction rejection of absolute, `..`, link, special-file, duplicate, and oversized
  entries;
- gzip/tar magic and malformed-stream rejection, including an oversized archive preseeded in
  the CAS so the network fetch is skipped;
- archive `strip_prefix`, `subdir`, and missing-`src_dir` behavior;
- explicit remote → remote and remote → local dependency graphs;
- repeated-process reproducibility test kept failing/disabled until upstream ordering is fixed;
- declared-output changes alter the action digest before deterministic caching is enabled;
- reproducibility fixtures covering both root-file ordering and nested-directory ordering;
- Smplx-version parity fixtures for every output-planning case.

## 17. Deferred questions

These questions do not block the local ruleset:

1. Should the stable package key be a readable escaped label, a label digest, or both?
2. Should a future adapter accept generated `.simf` files individually, or require a tree
   artifact carrying a complete package root?
3. Is upstream `--plan-json` worth contributing before implementing Anneal's output planner?
4. Should canonical manifests be materialized for native tooling, and which command owns that
   workflow?
5. Should remote source support remain `simplex_archive` sugar or wait for a generic
   `http_archive` rule?
6. When Smplx gains a lockfile with immutable revisions and hashes, does it replace explicit
   remote `deps`, or merely generate them through staged analysis?

None of these changes the central contract: **Simplex dependencies are typed Anneal package
edges; package origin is an implementation detail; `simplex build` receives a complete,
path-only, sealed package graph.**
